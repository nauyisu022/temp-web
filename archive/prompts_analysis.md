# Prompts.py 提示词分析报告

> 分析日期：2026-02-11
> 分析文件：`sft/sft_datagen/prompts.py`

---

## 一、整体架构

4 步串行流水线：

```
Step 1 用户画像生成 → Step 2 关键词提取 → Step 3 三层吸引力词构建 → Step 4 候选评估
```

设计思路合理——先理解用户、再提取原料、再组织结构、最后质量把关。但各步之间存在设计张力和潜在问题。

---

## 二、逐步分析

### Step 1: 用户画像生成（L11-L43）

#### 问题 1: 任务耦合度高，输出格式不够结构化

Step 1 把"用户画像分析"和"Schema 属性选择"两个本质不同的任务塞在一个 prompt 里。用户画像是开放式生成，Schema 选择是从候选集中筛选——认知模式完全不同。

输出格式是纯文本（`用户画像：...` / `选中Schema：...`），后续 Step 2 通过 `{user_profile}` 传入时，解析可靠性取决于 LLM 是否严格遵循格式。如果 LLM 在画像中多写了一行或换了格式，下游就可能解析失败。

**建议**：改为 JSON 输出，或者将两个任务拆分为两次调用。

#### 问题 2: 分析维度过多但约束不足

要求分析"人口统计、消费心理、使用场景、价值偏好"四个维度，但最终只要求"100字以内"的精简输出。四个维度的详细分析和 100 字的限制之间存在矛盾，LLM 要么压缩过度丢失信息，要么超出字数限制。

**建议**：要么放宽字数限制，要么减少分析维度，只保留对下游最有用的维度（如"核心需求"和"价值偏好"）。

#### 问题 3: Schema 选择缺乏边界情况处理

只说了"5-10 个"，但没有说明：
- 可用 Schema 本身少于 5 个怎么办？
- 原文信息极少的商品如何处理？
- 所有 Schema 都不太匹配时的降级策略？

**建议**：增加边界情况指导，如"如果可用属性不足 5 个，全部选择即可"。

---

### Step 2: 关键词提取（L50-L96）

#### 问题 4: 输入依赖 Step 1 的非结构化输出

`{user_profile}` 字段名叫"用户画像和选中 Schema"，但实际承载了两类信息。如果 Step 1 的输出格式稍有偏差，Step 2 就无法正确获取选中的 Schema 列表，导致不知道该提取哪些属性下的关键词。

**建议**：Step 1 改为 JSON 输出后，在 Step 2 中分别传入 `{user_profile}` 和 `{selected_schema}`。

#### 问题 5: "防止循环思考"指令有效性存疑

```
7. 【防止循环思考】
如果发现自己在重复考虑同一个问题：
    - 立即停止思考
    - 采用第一个想到的方案
    - 继续下一个属性
```

- 对于非 CoT 模型（如 GPT-4、Claude），这条指令没有意义，浪费 token
- 对于 CoT 模型（如 DeepSeek-R1、QwQ），"发现自己在重复考虑"这种元认知指令实际效果非常有限，模型很难真正执行自我监控
- 更好的做法是通过控制 `temperature`、`max_tokens` 或在系统 prompt 层面做约束

**建议**：移除此指令，改为在调用参数层面控制（如设置合理的 `max_tokens`）。

#### 问题 6: "保持原文表述"可能过于严格

对于多语言场景（繁体中文商品），原文可能有错别字、非标准表述。严格保持原文可能导致提取出质量不高的关键词。

**建议**：改为"优先保持原文表述，但可纠正明显的错别字"。

---

### Step 3: 三层吸引力词体系（L103-L295）— 核心 Prompt，问题最多

#### 问题 7: Prompt 过长，认知负荷极高

Step 3 约 **190 行**，包含：
- 格式要求
- 产品类型判断
- USP 独特性分析框架（3 个子框架，3 个级别）
- 三层结构的详细要求
- 输出要求（5 条）
- 两个完整的 JSON 示例
- 8 条特别注意事项

LLM 的注意力是有限的，当指令过多时，模型倾向于遵循最后看到的指令或最显眼的指令，而忽略中间的细节要求。

**建议**：
- 将"独特性分析"拆成独立的思考步骤（Step 2.5）
- 或者大幅简化指令，只保留核心规则 + 示例，用示例驱动而非规则驱动

#### 问题 8: 指令之间存在矛盾（P0 严重）

**矛盾 1 — brand 字段数量：**
- L166 说"brand, product_type 字段下只放 1 个词"
- 但示例中 `"brand":["優合", "Youha"]` 放了 2 个词（中英文品牌名）

**矛盾 2 — 创造新词 vs 排列组合：**
- L167 说"不要创造新词或组合词"
- L193 说"可以对词提取，进行一定排列组合"
- 这两条指令是矛盾的，"排列组合"本质上就是在创造新的组合

**矛盾 3 — 不要翻译 vs 提供英文版本：**
- L165 说"使用原文中的关键词，不要翻译"
- L221 说"每个值都提供原语言和英文版本"
- 虽然英文放在 `_en` 字段里，但这个逻辑需要更清晰地说明

**建议**：逐条审查指令，消除矛盾。brand 字段如果允许中英文品牌名共存，应明确说明。

#### 问题 9: "竞品差异分析"要求不现实（P0 严重）

```
2. 竞品差异分析
   - 该产品与同类竞品的关键差异是什么？
   - 哪些特性是该产品独有或领先的？
```

LLM 只看到了**当前这一个商品**的信息，没有任何竞品数据。要求它做"竞品差异分析"实际上是在鼓励幻觉——模型只能基于训练数据"猜测"竞品情况，这与"不要推断或创造"的原则直接冲突。

**建议**：
- 方案 A：移除竞品分析要求，改为"基于产品描述中的亮点进行独特性判断"
- 方案 B：如果确实需要竞品对比，在输入中提供同类竞品信息

#### 问题 10: 独特性分析框架过于学术化

三级独特性判定标准（高/中/低）虽然逻辑清晰，但对于 LLM 来说，这更像是一个供人类参考的分析框架，而不是一个可执行的指令。模型很难在没有竞品数据的情况下准确判定"该产品独有或极少竞品拥有"。

**建议**：简化为"优先提取产品描述中强调的独特卖点（如专利、认证、极限指标），避免提取通用特性"。

#### 问题 11: 示例偏向特定品类

两个示例都偏向母婴/家居用品（吸乳器配件、防撞角）。对于电子产品、服装、食品等品类，模型可能缺乏参考，导致输出质量不稳定。

**建议**：增加 2-3 个不同品类的示例（如电子产品、服装、食品），或者根据商品类目动态选择最相关的示例（few-shot selection）。

#### 问题 12: `_en` 双语输出增加出错概率

要求每个字段都输出原语言和英文版本，JSON 的 key 数量翻倍。特别是 USP 层的自定义属性名（如"材质安全"→"Material Safety"），翻译的一致性和准确性都是问题。

**建议**：考虑将翻译作为独立的后处理步骤，而非在同一个 prompt 中完成。

#### 问题 13: 数组 vs 字典的混合结构增加解析难度

- `top_attractiveness` 和 `unique_selling_points` 是字典（key 不固定）
- `long_tail_keywords` 是数组

USP 的 key 是 LLM 自由生成的（如"材质安全"、"耐用性能"），无法预先定义 schema，增加了下游解析和验证的复杂度。

**建议**：统一数据结构，或者为 USP 预定义一组候选 key（如"材质"、"性能"、"设计"、"认证"、"场景"），让 LLM 从中选择。

### Step 4: 候选评估（L302-L391）

#### 问题 14: 评估维度权重设计不合理

- **简洁性只有 5% 权重**，几乎可以忽略，单独列一个维度意义不大
- **吸引力 25% 权重**，但 Step 3 的任务本质是"提取"而非"创作"，强调吸引力可能鼓励模型选择那些添加了修饰语的候选，与"忠于原文"的原则冲突

**建议**：
- 移除简洁性维度，将其权重分配给准确性
- 降低吸引力权重，提高准确性权重（如准确性 40%、完整性 25%、结构性 20%、吸引力 15%）

#### 问题 15: 缺少加权计算的明确指令

给出了 5 个维度和各自的权重，但没有明确要求模型按权重计算加权分数。示例中的 `score: 3.85` 看起来像加权结果，但模型可能只是简单平均或凭感觉给分。

**建议**：明确要求"请按权重计算加权总分：总分 = 准确性×0.3 + 完整性×0.25 + 吸引力×0.25 + 结构性×0.15 + 简洁性×0.05"。

#### 问题 16: 评估者位置偏差（Position Bias）

5 个候选按固定顺序排列（候选 1 到候选 5），LLM 存在已知的位置偏差——倾向于选择靠前或靠后的候选。

**建议**：在调用时随机打乱候选顺序，并在 prompt 中明确提醒"不要受候选排列顺序影响"。

#### 问题 17: `score` 精度过高

示例中给出了两位小数的分数（如 3.85、4.20），这种精度暗示了虚假的精确性。LLM 的评分本质上是模糊的。

**建议**：改为整数或一位小数。

#### 问题 18: `confidence` 字段缺乏定义

输出中有 `"confidence": "high"`，但 prompt 中没有定义取值范围和判定标准。

**建议**：明确定义，如：
- `high`：最佳候选明显优于其他（分差 > 0.5）
- `medium`：最佳候选略优（分差 0.2-0.5）
- `low`：多个候选质量接近（分差 < 0.2）

## 三、跨步骤的系统性问题

### 问题 19: 信息传递的累积损失

Step 1 → Step 2 → Step 3 是串行的，每一步都依赖上一步的输出。如果 Step 1 的用户画像偏了，后续所有步骤都会偏。但整个流程没有任何**回溯或校验机制**。

**建议**：
- 在 Step 3 中重新传入原始商品信息（目前已做到），同时传入 Step 1 和 Step 2 的完整输出
- 考虑增加一个轻量级的校验步骤，检查最终输出是否与原始商品信息一致

### 问题 20: 语言混用问题

Prompt 指令用简体中文，但示例中大量使用繁体中文（"鉑金矽膠"、"儲存"、"軟管"）。这暗示目标市场是台湾/港澳，但 prompt 本身没有明确说明语言策略——提取的关键词应该保持原文语言（繁体），还是统一为某种语言？

**建议**：在 prompt 开头明确语言策略，如"提取的关键词保持商品原文语言，不做简繁转换"。

### 问题 21: 缺少错误处理和边界情况指导

整个流程没有考虑：
- 商品描述极短或为空的情况
- 商品信息包含大量无关内容（如店铺公告占 80%）的情况
- 非标准品类（如虚拟商品、服务类商品）的情况
- 商品信息是多语言混合的情况

**建议**：为每个 Step 增加边界情况处理指令，如"如果商品描述为空，仅基于标题和类目进行分析"。

### 问题 22: Token 效率问题

Step 3 的 prompt 本身约 2000+ tokens，加上商品描述和关键词，一次调用的输入可能达到 3000-5000 tokens。批量处理时成本显著。其中"USP 独特性分析框架"部分（约 60 行）对实际输出质量的提升可能不成比例。

**建议**：
- 精简 Step 3，移除"独特性分析框架"的详细描述，用简短规则 + 好示例替代
- 评估是否可以将 Step 1 和 Step 2 合并为一步

## 五、改进方向总结（v1 原始建议）

1. **结构化输出**：所有步骤统一使用 JSON 输出，确保下游可靠解析
2. **消除矛盾**：逐条审查 Step 3 的指令，消除互相矛盾的要求
3. **移除不现实要求**：删除竞品分析，改为基于产品描述本身的独特性判断
4. **精简 Step 3**：用"简短规则 + 多品类示例"替代"详细框架 + 少量示例"
5. **加强评估**：随机打乱候选顺序、明确加权计算、定义 confidence 标准
6. **增加鲁棒性**：为边界情况提供处理指导，明确语言策略
7. **考虑拆分翻译**：将 `_en` 翻译作为独立后处理步骤

## 六、v2 重构记录

> 重构日期：2026-02-11
> 新文件：`sft/sft_datagen/prompts_v2.py`

### 重构目标

从"忠实提取原文信息"转向"生成搜索友好的卖点关键词"，最大化搜索词覆盖，同时保持每个词都与产品相关。

### 核心设计原则

1. **语言约束**：所有关键词必须使用 title/description 中的语言（当地语言）
2. **搜索导向**：优先使用消费者会搜索的词，允许同义词扩展
3. **灵活度梯度**：brand 严格提取 > USP 允许同义词 > 长尾词最灵活
4. **精简指令**：用示例驱动，减少规则堆叠

### v1 → v2 逐步变更对照

#### Step 1: 用户画像 → 用户搜索意图

| 维度 | v1 | v2 |
|------|----|----|
| 任务定位 | 生成用户画像（人口统计、消费心理等） | 分析用户搜索意图（搜什么词、什么场景） |
| 输出格式 | 纯文本（"用户画像：... 选中Schema：..."） | 结构化 JSON（user_intent + search_word_types + selected_schema） |
| 对下游价值 | 画像信息对关键词提取引导有限 | 搜索意图直接指导关键词生成方向 |

解决的 v1 问题：#1（非结构化输出）、#2（维度过多约束不足）

#### Step 2: 关键词摘抄 → 搜索关键词提取与扩展

| 维度 | v1 | v2 |
|------|----|----|
| 核心约束 | "必须是原文中明确存在的内容"、"不要同义词替换" | "语义必须来自商品信息，但表述可以用同义词" |
| 数量 | 每个属性 1-3 个 | 每个属性 2-4 个，鼓励同义词变体 |
| 边界说明 | 无 | 明确的边界示例（✓ 保鮮袋 ✗ 抗菌） |
| 防循环思考 | 有（无效指令） | 移除 |

解决的 v1 问题：#4（非结构化输入）、#5（防循环思考无效）、#6（原文约束过严）

#### Step 3: 三层吸引力词 → 三层搜索关键词

| 维度 | v1 | v2 |
|------|----|----|
| 长度 | ~190 行 | ~110 行（精简 42%） |
| 竞品分析框架 | 有（30行，要求竞品差异分析） | 移除（无竞品数据，避免幻觉） |
| 独特性判定 | 三级框架（高/中/低，~20行） | 简化为优先级列表（3行） |
| USP 选择逻辑 | "避免通用特性（密封、BPA Free）" | "用户高频搜索的功能词即使常见也值得保留" |
| USP 数量 | 每个属性 1-2 个词 | 每个属性 2-3 个词 |
| 长尾词数量 | ≤4 个 | 5-6 个 |
| 长尾词约束 | "必须来自原文" | "允许别称、场景词、口语化表达" |
| 指令矛盾 | 3 处矛盾 | 消除 |
| 示例 | 2 个（吸乳器配件、防撞角） | 2 个（吸乳器配件、硅胶储存袋），示例中展示同义词扩展 |

解决的 v1 问题：#7（过长）、#8（指令矛盾）、#9（竞品分析不现实）、#10（框架过于学术化）

#### Step 4: 候选评估

| 维度 | v1 | v2 |
|------|----|----|
| 评估维度 | 5个（准确性30%、完整性25%、吸引力25%、结构性15%、简洁性5%） | 4个（语义准确性30%、搜索覆盖30%、完整性25%、结构规范15%） |
| 核心变化 | "吸引力"评估表达是否生动 | "搜索覆盖"评估是否覆盖多种搜索路径 |
| 加权计算 | 未明确要求 | 明确给出加权公式 |
| 位置偏差 | 未处理 | 增加"不要受排列顺序影响"提醒 |
| confidence | 未定义 | 明确定义三级标准 |
| 简洁性 | 5%（几乎无意义） | 合并移除 |

解决的 v1 问题：#14（权重不合理）、#15（缺少加权指令）、#16（位置偏差）、#18（confidence未定义）

### 风险与注意事项

1. **FactScore 可能下降**：v2 允许同义词，提取的词可能不出现在原文中，导致 FactScore 降低。但搜索覆盖率应该提升。需要在评估时权衡这两个指标。
2. **需要修改 processor.py 的 import**：将 `from prompts import ...` 改为 `from prompts_v2 import ...` 以启用 v2。
3. **JSON 输出兼容性**：v2 的 Step 1 改为 JSON 输出，processor.py 的 `_step2` 中 `{user_profile}` 传入的内容格式会变化，但 Step 2 的 prompt 已适配。
4. **v1 保留**：原 `prompts.py` 保留不动，方便 A/B 对比。
